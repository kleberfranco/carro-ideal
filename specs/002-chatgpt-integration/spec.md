# Feature Specification: ChatGPT Integration — Inteligência Artificial para Recomendação de Veículos

**Feature Branch**: `002-chatgpt-integration`

**Created**: 2026-06-13

**Status**: Planning

**Input**: TCC proposal "CARRO IDEAL AO SEU ALCANCE: Utilizando Inteligência Artificial para facilitar a escolha de Automóveis" — the core academic objective requires integrating the ChatGPT API as the recommendation engine.

---

## Vision and Scope

**Vision**: Replace the rule-based scoring algorithm with the ChatGPT API as the primary recommendation engine. The user fills the questionnaire, and the system sends the answers along with the vehicle catalog to ChatGPT, which returns a personalized ranked recommendation with natural language explanations. The existing scoring algorithm is kept as a silent fallback.

**Scope**:
- OpenAI API client (`net/http`, no external SDK)
- Prompt builder: questionnaire answers + vehicle catalog → structured prompt
- `AIService` that calls ChatGPT and parses the JSON response
- Graceful fallback to scoring algorithm when OpenAI is unavailable
- DB migration to persist the AI-generated summary
- Updated UI showing ChatGPT's natural language explanation
- Admin: vehicles description field used as ChatGPT context
- Tests for AIService with a mock HTTP client

**Out of Scope**:
- Streaming responses (SSE/WebSocket)
- Conversation history / multi-turn chat
- Fine-tuning or embeddings
- Cost tracking dashboard

## User Scenarios

### User Story 1 — ChatGPT-Powered Recommendation (P1)

**Narrative**: As a user, I complete the questionnaire and receive a ranking of vehicles chosen by ChatGPT, with a personalized explanation in Portuguese about why each car suits my profile.

**Acceptance Scenarios**:

1. **Given** a logged-in user submits questionnaire answers, **When** the OpenAI API is reachable, **Then** `POST /api/recommendations` responds with a ranked list where `Summary` is ChatGPT-generated and each item has a natural-language `Reason` from ChatGPT.
2. **Given** the same submission, **When** the OpenAI API returns an error or times out, **Then** the endpoint still responds 200 with recommendations generated by the scoring algorithm (fallback) and the `Summary` indicates "recomendação por score".
3. **Given** a previous recommendation, **When** the user accesses history, **Then** the AI-generated `Summary` is persisted and displayed correctly.
4. **Given** the admin has filled the vehicle `Description` field, **When** ChatGPT generates a recommendation, **Then** the description is included in the prompt as context for that vehicle.

### User Story 2 — Admin Vehicle Context (P2)

**Narrative**: As an admin, I fill the vehicle's description field to give ChatGPT richer context about each model (fuel type, target audience, highlights), improving recommendation quality.

**Acceptance Scenarios**:

1. **Given** a vehicle with a detailed description, **When** a recommendation is generated, **Then** the description appears in the ChatGPT prompt under that vehicle's entry.
2. **Given** a vehicle with no description, **When** a recommendation is generated, **Then** the prompt uses brand + model + category as fallback context.

## API Changes

### `POST /api/recommendations` — response unchanged structurally
The `summary` field now contains the ChatGPT-generated text (or scoring fallback text). A new `ai_summary` field carries the full ChatGPT narrative when available.

## Architecture

```
Handler
  └── RecommendationService.Generate(ctx, userID, answers)
        ├── QuestionnaireService.BuildProfile  (existing)
        ├── VehicleService.GetActive           (existing)
        ├── AIService.Recommend(ctx, answers, questions, vehicles)
        │     ├── PromptBuilder.Build(...)      → string
        │     ├── OpenAIClient.ChatComplete(ctx, prompt) → string
        │     └── parseAIResponse(json)         → AIRecommendation
        ├── [fallback] scoreVehicle()            (existing, if AI fails)
        └── recommendationRepo.Create(...)       (existing)
```

## Security & Reliability

- `OPENAI_API_KEY` never logged or exposed in responses
- 15s timeout on OpenAI HTTP call
- Fallback to scoring algorithm on any error (network, parse, API error)
- Rate limit already in place (users limited to 100 req/min) prevents API cost abuse

---

## Configuration

| Variable | Default | Description |
|----------|---------|-------------|
| `OPENAI_API_KEY` | — | Required in production; if empty, always uses fallback |
| `OPENAI_MODEL` | `gpt-4o-mini` | Model to use for recommendations |
| `OPENAI_TIMEOUT_SECONDS` | `15` | HTTP timeout for OpenAI calls |
